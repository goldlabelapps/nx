# TEFL Methodology for Technical Training

Teaching English as a Foreign Language (TEFL) to software developers requires adapting proven language teaching methodologies to technical contexts. Our approach combines communicative language teaching with task-based learning, all within authentic developer scenarios.

## Why TEFL Methodology Works for Developers

### Traditional English Classes Fail Developers

**Problems with General English**:
- Generic vocabulary irrelevant to work
- Academic writing instead of technical documentation
- Social conversations instead of professional communication
- Grammar-focused rather than communication-focused
- Disconnected from real work contexts

**What Developers Need**:
- Technical vocabulary and expressions
- Professional communication patterns
- Authentic work scenarios
- Immediate applicability
- Confidence in real work situations

### TEFL Principles Applied to Tech

**Communicative Language Teaching (CLT)**:
Focus on using language for real communication rather than just learning about language.

**Task-Based Learning (TBL)**:
Learn language by completing meaningful tasks, just like developers learn frameworks by building projects.

**Content and Language Integrated Learning (CLIL)**:
Learn English *while* learning technical content, not separately.

- Authentic communication contexts
- Real consequences and feedback

## Task-Based Learning for Developers

### The TBL Cycle

**1. Pre-Task** (Context and Preparation):
- Activate prior knowledge
- Introduce essential vocabulary
- Provide models and examples
- Set clear task objectives

**2. Task** (Real Communication):
- Authentic developer scenario
- Meaningful outcome required
- Language used as a tool
- Monitor without interrupting

**3. Planning** (Prepare to Report):
- Organize what to communicate
- Refine language choices
- Practice delivery
- Receive strategic support

**4. Report** (Share with Others):
- Present solutions or outcomes
- Receive peer feedback
- Compare approaches
- Build confidence

**5. Analysis** (Language Focus):
- Identify useful patterns
- Notice gaps and errors
- Learn specific forms
- Practice consciously

**6. Practice** (Controlled Activities):
- Drill useful patterns
- Memorize key phrases
- Build automaticity
- Prepare for next task

### Example Task Sequence

**Pre-Task**:
"Today we'll practice code review communication. Here are three PR comments—which are most effective and why?"

**Task**:
"Review this PR (actual code) and write 3 comments: one critical, one suggestion, one praise."

**Planning**:
"Revise your comments for clarity and tone. Use the diplomacy phrases we discussed."

**Report**:
"Share your comments. How did you balance directness with politeness?"

**Analysis**:
"Notice how modal verbs (could, might, would) soften suggestions. Let's collect examples."

**Practice**:
"Rewrite these direct comments using modal verbs and question forms."

## Error Correction Strategies

### When to Correct

**Immediate Correction** (Accuracy Focus):
- Practicing specific patterns
- Safety-critical communication
- Formal presentations
- Written documentation review

**Delayed Correction** (Fluency Focus):
- Free conversation
- Real-time problem-solving
- First drafts
- Brainstorming sessions

### How to Correct

**Recast** (Indirect):
Student: "I will fixing the bug"
Teacher: "Yes, you'll fix the bug. When will you fix it?"

**Clarification Request**:
"I didn't quite understand. Could you rephrase that?"

**Metalinguistic Clue**:
"There's something wrong with the verb tense there."

**Explicit Correction**:
"We say 'fix the bug,' not 'fixing the bug' in that sentence."

**Elicitation**:
"How do we say that in past tense?"

**Self-Correction**:
"Listen to what you said. Can you fix it?"

### Creating Correction Culture

- Errors are learning opportunities
- Self-correction is encouraged
- Peer correction builds community
- Focus on communication breakdowns, not every error
- Praise attempts and risk-taking

## Scaffolding Language Development

### Zone of Proximal Development

**Current Level** (Can do independently):
Simple written messages, basic code comments

**With Support** (Can do with help):
Complex PR descriptions, meeting participation

**Too Difficult** (Not yet accessible):
Conference presentations, complex negotiations

**Focus Here**: The "with support" zone

### Scaffolding Techniques

**Model-Practice-Produce**:
1. Show excellent examples
2. Practice with support
3. Create independently

**Sentence Starters**:
- "One approach might be..."
- "I'm concerned that..."
- "Could we consider..."

**Templates**:
PR description template, meeting agenda template, retrospective format

**Peer Collaboration**:
Practice together before solo performance

## Differentiation for Mixed Levels

- Facilitation roles

### Flexible Grouping

- Sometimes by task role, not level
- Dynamic based on activity
